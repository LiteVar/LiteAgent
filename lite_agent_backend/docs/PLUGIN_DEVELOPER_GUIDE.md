## Plugin Positioning
- The plugin service acts as a relay layer to integrate third-party platforms (DingTalk, WeChat Official Account, Website, etc.).
- Requests from third-party platforms go to the plugin service first, and the plugin calls LiteAgent main service Agent APIs to complete business logic.
- The configuration page is rendered by the LiteAgent frontend, and configuration data is provided by the plugin service.
- The configuration page binds the current connector to an agent and stores extra settings (for example, appId and secret for an official account).

## Plugin and Connector Relationship
- A plugin is a service-level capability, and a connector is a configuration instance for a specific business scenario.
- One plugin can have multiple connectors. Each connector has a unique `connectorId` generated by the main service.
- The plugin should store configuration and status using `connectorId` as the primary key; connectors do not affect each other.
- Connector status is controlled by `/plugin/status`. Offline connectors should not process third-party requests.
- Connector deletion is handled by `/plugin/delete`.

## Plugin Service Requirements
- Run as an HTTP service inside the container and expose a single service port.
- The plugin service must listen on a fixed port `8888` (container internal port). The host port is mapped by the runner.
- Implement `/plugin/health`, `/plugin/schema`, `/plugin/data`, `/plugin/config`, `/plugin/status`, `/plugin/delete`.
- Directory and permission rules:
  - Persistent data directory: `/data/lite-agent-plugin` (only this directory is writable and used to store important data).
  - Package directory: `/opt/lite-agent-plugin/app` (read-only; do not write packages to the data directory to avoid upgrade rollback).
  - Only `/data/lite-agent-plugin` and `/tmp` are writable; all other paths must be read-only.
  - `/tmp` uses `tmpfs` with a size limit of `64MB`.
- The plugin must persist its own configuration data (the main service does not store plugin config).
- The plugin must maintain connector online/offline status; offline connectors must not handle third-party requests.

## Pairing and Authentication with the Main Service
When the container starts, `encryptedKey` is written to `/data/lite-agent-plugin/key/encryptedKey.txt`. The plugin reads and decrypts it for HMAC signing.

Request timeouts:
- Connect timeout: 1s
- Read timeout: 5s
- No retries

### Key File
- File path: `/data/lite-agent-plugin/key/encryptedKey.txt`
- File content: AES+Base64 encrypted string (`encryptedKey`)
- Decryption steps:
  1) Base64 decode `encryptedKey`
  2) AES decrypt to get `sharedKey`
  3) Save `sharedKey` for HMAC usage
- AES key (hardcoded, must match the main service):
  - `UdYbRx48hZQR8cys`

AES details:
- Algorithm/Mode/Padding: `AES/ECB/PKCS5Padding`
- Key encoding: UTF-8
- IV: none
- Ciphertext encoding: Base64

### HMAC Authentication
All plugin APIs (except `/plugin/health`) called by the main service must include:
- `X-TS`: Millisecond timestamp
- `X-Nonce`: Random nonce
- `X-Sign`: HMAC signature (`HMAC_SHA256(sharedKey, canonical)`)

Canonical string format:
```
METHOD\nPATH\nTS\nNONCE\nQUERY\nBODY_SHA256\nCONTENT_TYPE
```
Notes:
- METHOD is uppercase HTTP method (GET/POST).
- PATH is the request path (without query).
- QUERY is the normalized query string (empty when no query).
- BODY_SHA256 is the SHA256 of the raw request body bytes (empty string for empty body).
- CONTENT_TYPE is the raw `Content-Type` header value (empty when not present).

Validation rules:
- `X-TS` must be within the allowed time window (recommended 10 seconds).
- `X-Nonce` deduplication is up to the plugin.
- `X-Sign` must match the HMAC of the canonical string.

Query normalization rules:
- Sort by key in lexicographical order (full sort, not just first letter).
- For the same key, sort values in lexicographical order.
- Join as `key=value` pairs with `&`.
- URL-encode keys and values; empty values use `key=`.

## API Definitions

Parameter rules:
- `GET` uses query parameters.
- `POST` uses JSON request body (`Content-Type: application/json`).

### Health Check
`GET /plugin/health?encryptedText=xxx`

Purpose:
- Check plugin service availability.

Request parameters:
- `encryptedText`: Encrypted string

Response example:
```json
{
  "plainText": "random-text"
}
```

Notes:
- `encryptedText` uses the same AES rules as the key file.
- Decryption: Base64 decode -> AES decrypt (AES/ECB/PKCS5Padding, UTF-8 key).
- The plugin returns the original `plainText`.

### Get Schema
`GET /plugin/schema`

Purpose:
- Return the JSON Schema required by the frontend configuration form.

Request parameters: none

Response example:
```json
{}
```

Field notes:
- JSON schema data for the form

### Get Data
`GET /plugin/data?connectorId=xxx`

Request parameters:
- `connectorId`: connectorId

Response example:
```json
{}
```

Notes:
- Used to populate configuration data for the config page.
- The data structure matches the schema returned by `/plugin/schema`.
- Return an empty object when no data exists.

### Save Config
`POST /plugin/config`

Request body example:
```json
{
  "connectorId": "xxx",
  "data": {}
}
```

Response example:
```json
{
  "success": true,
  "errors": []
}
```

Notes:
- The plugin must persist `data`.
- `errors` is used to return validation errors.
- `data` must match the schema returned by `/plugin/schema`.

### Set Connector Status
`POST /plugin/status`

Request body example:
```json
{
  "connectorId": "xxx",
  "offline": true
}
```

Response example:
```json
{
  "success": true
}
```

Notes:
- The plugin must update and persist the status based on `offline`.
- `offline=true` means offline, `offline=false` means online.

### Delete Connector
`POST /plugin/delete`

Request body example:
```json
{
  "connectorId": "xxx"
}
```

Response example:
```json
{
  "success": true
}
```

Notes:
- The plugin must delete the connector configuration and status.
- Must be idempotent: repeated deletes should not error.

### Get User Access Statistics
`POST /plugin/analyze`

Purpose:
- Aggregate user access data for a time range, for dashboards and analytics.

Request body example:
```json
{
  "connectorId": "xxx",
  "startTime": "2025-12-01 00:00:00",
  "endTime": "2025-12-31 23:59:59"
}
```

Response example:
```json
{
  "statistics": {
    "activeUserCount": 12,
    "averageConversationRounds": 3.5,
    "averageSessionsPerActiveUser": 1.4,
    "totalTokenConsumption": 1621
  },
  "activeUserTrend": [
    {
      "time": "2025-12-01 00:00:00",
      "activeUserCount": 1
    }
  ],
  "conversationRoundsDistribution": [
    {
      "roundsRange": "1-2 rounds",
      "count": 8
    }
  ],
  "sessionDistribution": [
    {
      "sessionRange": "1 time",
      "userCount": 9
    }
  ]
}
```

Notes:
- `startTime`/`endTime` are time strings, format recommended `yyyy-MM-dd HH:mm:ss`.
- `statistics` contains summary data: active user count, average rounds, average sessions per active user, total token usage.
- `activeUserTrend` is the time series; `time` is a string time.
- `conversationRoundsDistribution` is the rounds distribution; `roundsRange` is the range label.
- `sessionDistribution` is the session count distribution; `sessionRange` is the range label.

## Error Handling
Error handling rules:
- Auth failure (HMAC/time window/nonce): HTTP 401, returns `{ "code": "UNAUTHORIZED", "message": "..." }`
- Invalid params: HTTP 400, returns `{ "code": "INVALID_PARAM", "message": "..." }`
- System error: HTTP 500, returns `{ "code": "INTERNAL_ERROR", "message": "..." }`
- `/plugin/config` validation failure: HTTP 500, returns `{ "code": 500, "message": '.....' }`

Error codes (fixed values):
- `UNAUTHORIZED`
- `INVALID_PARAM`
- `INTERNAL_ERROR`

## Packaging
Only a Docker image file is required. Uploading follows the platform requirements.

Image directory conventions:
- Package directory is fixed at `/opt/lite-agent-plugin/app`. The app must be placed here in the image.
- Persistent data directory is fixed at `/data/lite-agent-plugin`. The plugin should only write data to this directory.
- Do not write packages or runtime data to other paths, to avoid failures under read-only rootfs or upgrade rollback.

Example:
```bash
# Build image
docker build -t dingtalk-robot:1.0.0 .
# Export image
docker save -o image.tar dingtalk-robot:1.0.0
```
